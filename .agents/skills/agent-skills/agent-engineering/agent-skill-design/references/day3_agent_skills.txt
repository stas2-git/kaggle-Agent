# Agent Skill Design Reference: Operational Design Rules

Use this reference when creating or refactoring a skill from source material.

## A Good Skill

- Has one job.
- Has a description that acts like a routing classifier.
- Gives a concrete procedure, not generic advice.
- Loads detailed context only when needed.
- Uses scripts for deterministic repeated operations.
- Includes eval prompts or pass/fail checks for future improvement.

## What Goes Where

| Material | Put it in |
|---|---|
| Trigger and purpose | `SKILL.md` frontmatter description |
| Core workflow | `SKILL.md` body |
| Long background, schemas, checklists, examples | `references/` |
| Deterministic repeated commands or transforms | `scripts/` |
| Templates, images, starter files, reusable output | `assets/` |

## Reference Refactoring Rule

When extracting from a whitepaper or long document:

1. Identify the skill's job.
2. Pull only the concepts that change the agent's behavior for that job.
3. Convert concepts into checklists, decision rules, templates, or eval prompts.
4. Remove narrative, motivation, and repeated background.
5. Keep the original source in central reference material, not duplicated inside every skill.

## Trigger Validation Prompts

For every skill, write:

- Should trigger: a direct user request.
- Should trigger edge case: same job with ambiguity or constraints.
- Should not trigger: adjacent work handled by another skill or no skill.

## Quality Smell Tests

- If a reference can be summarized as "be a better agent," rewrite it.
- If the agent must read hundreds of lines before acting, split or shorten it.
- If a skill has unrelated verbs in its job, split it.
- If two skills trigger on the same prompt, sharpen descriptions or route through a parent.
- If a deterministic step appears repeatedly, make it a script.
