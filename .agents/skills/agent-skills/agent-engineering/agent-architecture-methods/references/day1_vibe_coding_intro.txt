# Agent Architecture Reference: Harness Design

Use this reference when turning an agent idea into a buildable system.

## Architecture Frame

Design the agent as a harness, not just a prompt:

- Model: what reasoning capability is needed?
- Instructions: what stable behavior must always apply?
- Context: what task-specific information is loaded, and when?
- Tools: what external actions or data does the agent need?
- State: what must persist across turns, jobs, or events?
- Guardrails: what must be blocked, checked, or approved?
- Observability: how will runs be inspected?
- Evals: how will quality and safety be measured?

## Design Questions

- What does success look like to the user?
- What should be deterministic code instead of model judgment?
- What actions require human approval?
- What data or tools should be read-only at first?
- What failure mode would be most costly?
- What evidence proves the agent did the right thing?

## Common Architecture Mistakes

- Treating a prompt as the whole system.
- Giving broad tools before defining authority boundaries.
- Using static context as a database.
- Building multi-agent flow when one agent plus tools is enough.
- Skipping tests/evals because the demo works.
- Optimizing generation speed while ignoring review and rollback.
