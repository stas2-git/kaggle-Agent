# Agent Protocol Selector Reference: Decision Table

Use this reference to choose the smallest useful primitive for a capability.

## Selection Table

| User need | Choose | Because |
|---|---|---|
| Reusable know-how or workflow | Skill | Loads procedural memory only when relevant |
| Deterministic repeated step | Script | More reliable and cheaper than re-prompting |
| External API, database, file system, or SaaS access | MCP/tool | Gives structured access with schemas and auth |
| Specialist must negotiate, plan, or own a multi-turn outcome | A2A agent | A function call is too bounded |
| User needs to review, approve, or edit structured state | A2UI-style UI | Safer than arbitrary generated interface code |
| Purchase, payment, mandate, or commerce authorization | AP2/UCP-style flow | Requires explicit authority and auditability |
| Stable project convention | `AGENTS.md` | Always-on local rule |
| One-off local implementation | Application code | No reusable agent capability needed |

## Build-Versus-Buy Checks

- If official or managed tooling exists, prefer it over a custom connector.
- If the task only needs a stable command, use a script.
- If the task needs broad external authority, narrow the scope before connecting it.
- If a tool needs clarification and memory, consider an agent handoff instead.

## Safety Checks

- Keep tool count small.
- Start read-only.
- Pin and audit third-party servers/skills.
- Avoid giving a general agent direct transaction authority.
- Log delegated actions and approval decisions.
